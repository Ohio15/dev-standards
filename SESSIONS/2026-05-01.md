## Premise corrections (high-value)

The 8-agent research wave surfaced two MAJOR plan-document errors that would have driven incorrect Sentinel rotation work.

### 1. Sentinel "binary bake-in" claim is empirically false

Plan doc `CREDENTIAL_ROTATION_PLAN.md:34` and `:92` asserts API keys and enrollment tokens are baked into compiled agent binaries, requiring binary rebuild on rotation.

Per agent **B3** report: scan of all 7 EXE artifacts at commit `bad511d^` (the last green agent build) shows:
- `0 occurrences` of API_KEY substring (`55ccf1fd...`)
- `0 occurrences` of ENROLLMENT_TOKEN substring (`40addfff...`)

ENROLLMENT_TOKEN is actually delivered via `%%ENROLLMENT_TOKEN%%` sed-substitution into JSON configs inside `.deb`, `.pkg`, and `.spk` packaging artifacts at install time ‚Äî NOT PE binary patching. `git filter-repo --replace-text` is therefore SAFE to run against the repo without invalidating any released binary.

### 2. MigrateStaticAPIKey shim does not "persist bcrypt -> still validates"

Plan claims the shim writes a bcrypt hash of the static key into the `api_keys` table on first call, allowing the legacy static key to keep validating via the DB path even after env-var removal. Per agent **B4** report:

- The shim has **zero callers** in source (`grep -R MigrateStaticAPIKey` returns only the definition site). It is dead code; nothing invokes it.
- Even if invoked, `ValidateKey` enforces a format gate at `api_key_manager.go:154` requiring `sk_live_` prefix. The static key (`55ccf1fd...`, no prefix) is rejected before any DB lookup runs.

Both invariants in the plan are wrong. D1/D2 (delete vs revoke) handed to Ron ‚Äî see ledger below.
